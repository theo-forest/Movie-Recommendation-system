# Movie Recommendation System

A production-grade movie recommendation system built on the **MovieLens** dataset.
It implements three complementary recommendation strategies — collaborative filtering,
content-based filtering, and a weighted hybrid — served through a **FastAPI** backend,
explored through a **Jupyter notebook**, monitored in real time, and deployable with
**Docker**.

## 2. The Dataset

### MovieLens Small (ml-latest-small)

The dataset is published by the **GroupLens** research lab at the University of Minnesota
and is a standard benchmark in recommender systems research.

| File | Rows | Description |
|------|------|-------------|
| `movies.csv` | 9 742 | Movie metadata: `movieId`, `title` (with year), `genres` (pipe-separated) |
| `ratings.csv` | 100 836 | User ratings: `userId`, `movieId`, `rating` (0.5–5.0, half-star steps), `timestamp` |
| `tags.csv` | 3 683 | Free-text tags applied by users: `userId`, `movieId`, `tag`, `timestamp` |
| `links.csv` | 9 742 | Cross-references to IMDB and TMDB identifiers |

**Key statistics after loading:**

- **610** unique users
- **9 724** unique rated movies (not all 9 742 movies have received ratings)
- **100 836** ratings → matrix density ≈ **1.7 %** (very sparse)
- Ratings follow a roughly normal distribution centred around **3.5**
- Strong **long-tail**: the top 10 % of movies account for ~60 % of all ratings

The sparsity is the central challenge of this problem. Most users have rated fewer than
200 movies out of nearly 10 000 available — the system must generalise from very few
observations per user.

## 3. Data Preprocessing

### 3.1 Loading (`src/data_loader.py`)

`load_movies()` enriches the raw CSV with:
- **`year`** — extracted from the title string `"Toy Story (1995)"` via regex
- **`title_clean`** — title without the year suffix
- **`genres_list`** — the pipe-separated genres split into a Python list

`load_ratings()` converts Unix timestamps to `datetime` objects so the data can be
sorted and split chronologically.

### 3.2 Cold-start filtering

`filter_min_interactions(ratings, min_user_ratings=5, min_movie_ratings=5)` removes:
- Users who have rated fewer than 5 movies
- Movies that have received fewer than 5 ratings

**Why?** Items and users with very few interactions produce unreliable latent factor
estimates in matrix factorisation and distort evaluation metrics. After filtering,
100 836 → **90 274** ratings remain across 610 users and 3 650 movies.

### 3.3 Train / test split (`train_test_split_temporal`)

The ratings are sorted by timestamp and the last **20 %** of interactions form the
test set.

**Why temporal and not random?**
A random split would allow the model to see future information — for example, a user's
rating from 2015 could be in the training set while an earlier rating from 2010 is in
the test set. In production the model will always be trained on past data and evaluated
on future data, so the temporal split is the only realistic evaluation strategy.

### 3.4 User-item matrix (`build_user_item_matrix`)

A dense pivot table of shape `(n_users, n_movies)` is constructed from the training
ratings. Cells with no rating are `NaN`. For the SVD step this table is:

1. **Mean-centred per user** — each user's average rating is subtracted from their
   row. This removes the *rating-scale bias* (some users consistently rate everything
   4/5; others are harsher critics). The model then learns *preference deviations*
   from each user's personal baseline.
2. **NaN-filled with 0** — missing entries are treated as "no preference deviation"
   (the user's mean), which is the standard assumption for implicit/explicit CF.
3. **Converted to a CSR sparse matrix** — storing only the non-zero entries saves
   memory and accelerates the decomposition.

## 4. Models — Why These Three?

The three models cover the two canonical families of recommendation approaches and
address each other's weaknesses.

---

### 4.1 Collaborative Filtering (SVD)

**File:** `src/models.py` — class `CollaborativeFilter`

#### How it works

Collaborative filtering answers: *"Users who behaved similarly to you in the past
liked X — you will probably like X too."* It requires no knowledge of what the items
actually are; it only needs the pattern of ratings across users.

The implementation uses **Truncated SVD** (Singular Value Decomposition) from
`sklearn.decomposition.TruncatedSVD` to factorise the mean-centred user-item matrix
`M` into:

```
M ≈ U · Σ · Vᵀ
```

Where:
- **U** `(n_users, k)` — each row is a user's position in a k-dimensional latent space
- **Σ** `(k,)` — importance of each latent dimension (absorbed into U)
- **Vᵀ** `(k, n_movies)` — each column is a movie's position in the same space

The `k = 50` latent dimensions are interpretable as abstract "taste dimensions" (e.g.
"likes action", "prefers arthouse", "favours 90s films") that the algorithm discovers
automatically from the data.

**Prediction:** For user `u` and movie `i`:
```
predicted_rating(u, i) = dot(U[u], V[i]) + mean_rating[u]
```

The mean rating is added back after reconstruction to restore the original scale.

**Recommendation:** Compute predicted ratings for all unseen movies, sort descending,
return top-N. Already-seen movies are masked to `-∞` before sorting.

#### Why SVD and not other CF methods?

| Method | Pro | Con |
|--------|-----|-----|
| User-user KNN | Intuitive | Doesn't scale to millions of users |
| Item-item KNN | Stable, explainable | Memory-intensive for large catalogues |
| **SVD (matrix factorisation)** | Compact representation, handles sparsity well, fast at query time | Black box, cold-start problem |
| ALS (Alternating Least Squares) | Better for implicit feedback | Harder to implement, requires `implicit` library |
| Neural CF / NCF | Captures non-linear patterns | Needs much more data and compute |

SVD is the right choice here: it is well-understood, fast to train on 90K ratings
(< 1 second), and produces meaningful latent representations. The Netflix Prize
(2009) was largely won by matrix factorisation variants.

---

### 4.2 Content-Based Filtering (TF-IDF)

**File:** `src/models.py` — class `ContentBasedFilter`

#### How it works

Content-based filtering answers: *"You liked movie X — here are movies with similar
content."* It requires no user history; it only needs the item attributes.

Each movie is represented as a **TF-IDF vector** over a combined text field:
```
features = genres_text + " " + title_words
```

For example, *"Toy Story (1995)"* becomes:
```
"Adventure Animation Children Comedy Fantasy Toy Story"
```

**TF-IDF** (Term Frequency — Inverse Document Frequency) weights each token by how
often it appears in this movie's description relative to how common it is across
all movies. Rare genre combinations (e.g. "Animation|Adventure|Fantasy") get higher
weights than ubiquitous ones (e.g. "Drama").

With `ngram_range=(1, 2)`, both unigrams (`"Action"`) and bigrams (`"Action Adventure"`)
are indexed, capturing co-occurring genre pairs as features.

**Recommendation:** Compute the **cosine similarity** between the query movie's
TF-IDF vector and all other movies' vectors. Return the top-N most similar.

Cosine similarity is used instead of Euclidean distance because it is invariant to
document length — a short two-genre description and a long five-genre description
can still be meaningfully compared.

#### Why TF-IDF and not sentence embeddings?

The original scaffold used `sentence-transformers` (BERT-style embeddings). TF-IDF
was chosen for the content model because:

1. **Interpretability** — the similarity is directly traceable to shared genre tokens
2. **Speed** — TF-IDF fit + cosine similarity for 9 742 movies takes ~100 ms vs
   several seconds for BERT encoding
3. **Sufficient signal** — genre and title are strong predictors of content similarity;
   deep semantic embeddings add little when you don't have plot summaries
4. **No GPU required** — keeps the system deployable on any machine

Sentence embeddings would be the right upgrade if the dataset included plot synopses
or user reviews.

### 4.3 Hybrid Recommender

**File:** `src/models.py` — class `HybridRecommender`

#### How it works

The hybrid model combines both approaches with a **weighted linear blend**:

```
hybrid_score(movie) = 0.7 × norm(CF_score) + 0.3 × norm(CBF_score)
```

Both score lists are **min-max normalised** to [0, 1] before blending so that the
different score scales (predicted ratings vs cosine similarities) don't dominate
each other.

**For a given user:**
1. CF produces predicted ratings for all unseen movies
2. The user's top-rated seen movie seeds the content-based filter
3. Scores from both are normalised and blended
4. The union of both candidate sets is ranked by hybrid score

**For a given movie (no user context):** falls back to pure content-based similarity.

#### Why a hybrid?

Each model has a specific weakness that the other compensates for:

| Weakness | Caused by | Fixed by |
|----------|-----------|----------|
| CF popularity bias | Users tend to watch popular movies → SVD over-represents them | CBF injects content-diverse candidates |
| CF cold-start (new user) | No rating history → no latent vector | CBF uses item attributes only |
| CBF filter bubble | Only recommends items similar to what you already know | CF brings in serendipitous user-based diversity |
| CBF ignores social signal | Doesn't know which movies people actually liked | CF contributes the collective taste signal |

The **70 / 30 split** favours CF because it captures actual user preference signals
more directly, but the 30 % CBF contribution meaningfully increases catalogue coverage
(more diverse long-tail movies appear in recommendations).

## 5. Evaluation

**File:** `src/evaluation.py`

Two complementary evaluation layers are used:

### 5.1 Rating Prediction Quality

Measures how accurately the model predicts the exact numeric rating a user would give.

| Metric | Formula | Interpretation |
|--------|---------|----------------|
| **RMSE** | `√(mean((y_true - y_pred)²))` | Penalises large errors heavily. Lower = better |
| **MAE** | `mean(|y_true - y_pred|)` | Average absolute error. More robust to outliers |

**Baseline comparison:** Predicting the global mean rating (≈ 3.5) achieves
RMSE ≈ 1.06. The CF model should beat this baseline to be considered useful.

Achieved results on MovieLens Small: **RMSE ≈ 1.10, MAE ≈ 0.84**.

### 5.2 Ranking Quality (Top-K metrics)

Rating prediction quality is necessary but not sufficient — in practice users receive
a ranked list of N movies, not a predicted score. These metrics evaluate the quality of
that ranked list.

| Metric | Measures |
|--------|----------|
| **Precision@K** | Of the K items shown, what fraction are relevant? |
| **Recall@K** | Of all relevant items, what fraction appear in the top K? |
| **NDCG@K** | Are relevant items ranked near the top? (rewards early hits more) |
| **HitRate@K** | Did the user find at least one relevant item in the top K? |

"Relevant" = rated ≥ 3.5 in the test set.

**Why both layers?** A model that predicts ratings accurately may still produce poor
ranked lists (and vice versa). NDCG@K is the metric most correlated with user
satisfaction in deployed systems.

### 5.3 Catalogue Coverage

```
coverage = (unique movies recommended to any user) / total_movies
```

A low coverage (< 5 %) means the model is stuck in a popularity loop — the same
blockbuster movies are recommended to everyone. Coverage is a diversity proxy.

## 6. Monitoring

**File:** `src/monitoring.py` — class `ModelMonitor`

In production, models degrade silently over time as user preferences shift and the
catalogue grows. The `ModelMonitor` class addresses this with three mechanisms:

### 6.1 Performance Tracking

`log_performance(y_true, y_pred, label)` computes RMSE and MAE and appends them to
an in-memory log with timestamps. An alert fires if RMSE exceeds a configurable
threshold (`alert_rmse_threshold`, default 1.2).

### 6.2 Data Drift Detection

`detect_data_drift(reference, current, alpha=0.05)` runs a
**Kolmogorov-Smirnov (KS) two-sample test** between two rating distributions (e.g.
ratings from month 1 vs month 2).

The KS test is non-parametric — it makes no assumptions about the distribution shape
and is sensitive to any change in the distribution (mean, variance, skewness). A
p-value < 0.05 signals that the two samples are unlikely to come from the same
distribution, indicating **concept drift** — users may be rating differently, a new
genre may have surged in popularity, or data quality may have changed.

**Why not PSI (Population Stability Index)?** The KS test is more appropriate here
because we are comparing continuous distributions (rating values 0.5–5.0 in 0.5 steps).
PSI is better suited to binary or categorical features.

### 6.3 Recommendation Logging

`log_recommendation_event(user_id, recommended, scores)` records every recommendation
served, enabling offline computation of:
- **Catalogue coverage** over time
- **Average list length**
- Per-movie recommendation frequency (to detect runaway popularity bias)

All monitor state can be exported to JSON with `to_json(path)` for integration with
external dashboards.

## 7. API

**File:** `src/api.py`  
**Framework:** FastAPI  
**Start:** `uvicorn src.api:app --reload --port 8000`

Models are loaded once at startup via the `lifespan` context manager and held in
module-level globals. All routes are synchronous (`def`, not `async def`) since the
model inference calls are CPU-bound.

### Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/recommend` | Get movie recommendations |
| `GET` | `/movie/{movie_id}` | Fetch metadata for a single movie |
| `GET` | `/movies/search?q=<query>` | Full-text search on movie titles |
| `GET` | `/metrics` | Retrieve last saved evaluation metrics |
| `GET` | `/health` | Liveness check with model load status |

### POST `/recommend`

**Request body:**

```json
{
  "user_id": 1,
  "n": 10,
  "model_type": "hybrid"
}
```

OR for item-based (similar movies):

```json
{
  "movie_id": 1,
  "n": 10
}
```

**Fields:**

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `user_id` | int | null | Get personalised recommendations for this user |
| `movie_id` | int | null | Get movies similar to this one |
| `n` | int | 10 | Number of results (1–50) |
| `model_type` | str | "hybrid" | `"hybrid"` / `"collaborative"` / `"content"` |

**Response:**

```json
{
  "recommendations": [
    {
      "movie_id": 527,
      "title": "Schindler's List (1993)",
      "genres": "Drama|War",
      "year": 1993.0,
      "score": 0.8241
    }
  ],
  "explanation": "Hybrid (CF 70% + CBF 30%) recommendations for user 1.",
  "model_used": "hybrid"
}
```

**Error codes:**

| Code | Meaning |
|------|---------|
| 400 | Neither `user_id` nor `movie_id` provided, or invalid `model_type` |
| 404 | No recommendations could be generated |
| 503 | Requested model not loaded (run `src/train.py` first) |

### GET `/health`

```json
{
  "status": "ok",
  "models": {
    "collaborative_filter": true,
    "content_filter": true,
    "hybrid_recommender": true
  },
  "movies_loaded": true
}
```

## 8. Streamlit UI

**File:** `app/streamlit_app.py`  
**Start:** `streamlit run app/streamlit_app.py`  
**Access:** `http://localhost:8501`

The UI communicates exclusively with the FastAPI backend via HTTP — it contains no
model logic itself.

### Features

**Sidebar:**
- Mode selector: *By User ID* or *By Movie*
- Model picker: `hybrid`, `collaborative`, `content`
- Result count slider (5–20)
- Live API health status showing which models are loaded

**Main area:**
- *By User ID mode:* enter a user ID (1–610) → the selected model returns
  personalised recommendations ranked by score, with title, genre tags, and score
- *By Movie mode:* search by keyword → select a movie from the results →
  content-based similar movies are returned
- Collapsible **Model Evaluation Metrics** panel showing RMSE, MAE,
  Precision@K etc. fetched from the `/metrics` endpoint

## 10. Docker & Deployment

### Dockerfile

```dockerfile
FROM python:3.9-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY src/ ./src/
COPY app/ ./app/
COPY data/ ./data/
COPY config.json ./
RUN python -m src.train        # models are trained at build time
EXPOSE 8000
CMD ["uvicorn", "src.api:app", "--host", "0.0.0.0", "--port", "8000"]
```

**Key decision — train at build time:** Models are baked into the Docker image. This
means the container starts instantly without needing a training phase and guarantees
reproducibility — every deployment uses exactly the same model weights.

The downside is that retraining requires rebuilding the image. For a production
system with frequent retraining, a better pattern is to mount a `models/` volume
(already in `docker-compose.yml`) and retrain via a separate scheduled job.

### docker-compose.yml

Two services:

| Service | Port | Role |
|---------|------|------|
| `api` | 8000 | FastAPI backend; `models/` directory mounted as a volume |
| `streamlit` | 8501 | Streamlit UI; depends on `api` (starts after it is healthy) |

A **healthcheck** is configured on the `api` service (`GET /health`) so Docker knows
when the backend is ready before starting Streamlit.

**Start everything:**
```bash
docker-compose up --build
```

**Access:**
- API docs (Swagger UI): `http://localhost:8000/docs`
- Streamlit UI: `http://localhost:8501`

## 11. Quick Start

### Without Docker (recommended for development)


## 13. Configuration

All tuneable parameters live in `config.json`:

```json
{
  "data": {
    "raw_path": "data/raw/ml-latest-small",
    "processed_path": "data/processed"
  },
  "models": {
    "models_dir": "models",
    "collaborative_filter": {
      "n_factors": 50,
      "random_state": 42
    },
    "content_filter": {
      "max_features": 5000
    },
    "hybrid": {
      "cf_weight": 0.7,
      "cbf_weight": 0.3
    }
  },
  "training": {
    "test_ratio": 0.2,
    "min_ratings_per_user": 5
  },
  "api": {
    "host": "0.0.0.0",
    "port": 8000,
    "default_n_recommendations": 10
  }
}
```

**Key parameters to tune:**

| Parameter | Effect |
|-----------|--------|
| `n_factors` | Higher = more expressive CF model, but slower and more prone to overfitting. Try 20–200 |
| `cf_weight` / `cbf_weight` | Shift the balance between personalisation and content diversity. Must sum to 1.0 |
| `max_features` | Number of TF-IDF vocabulary terms. Higher = richer content representation |
| `test_ratio` | Fraction of interactions held out for evaluation |
| `min_ratings_per_user` | Raising this reduces the cold-start noise but shrinks the training set |
